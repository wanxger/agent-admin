## Context

当前 CLI 工具支持通过配置文件（`aa.yaml`）和命令行参数（`--cwd`, `--agent`）来配置行为。配置文件的优先级规则已在文档中说明，但在实际使用中可能存在不一致的情况，需要通过代码验证来确保文档描述的行为与实际实现一致。

## Goals / Non-Goals

**Goals:**
- 确保配置文件和命令行参数的优先级规则在代码层面得到验证
- 提供清晰的错误提示，当配置文件和命令行参数存在冲突时
- 完善文档，明确说明配置层级关系
- 添加配置验证逻辑，确保配置的完整性

**Non-Goals:**
- 修改现有的配置优先级规则（保持现有行为）
-`添加新的配置选项（仅验证现有配置）`

## Decisions

- **配置验证时机**：在 CLI 启动时、解析完所有配置后进行验证，而不是在解析过程中
  - **理由**：这样可以同时看到所有配置源（文件、命令行、环境变量），便于检测冲突
  - **替代方案**：在解析每个配置源时立即验证 - 但无法检测跨源冲突

- **冲突处理策略**：对于已知冲突（如 `--cwd` vs 配置文件的 `cwd`），命令行参数优先（保持现有行为），但输出警告信息
  - **理由**：保持向后兼容，不破坏现有用户的工作流
  - **替代方案**：抛出错误拒绝启动 - 但这可能影响现有用户的脚本

- **验证规则实现**：集中在一个配置验证器模块中，便于维护和测试
  - **理由**：验证逻辑集中管理，避免重复代码，便于单元测试
  - **替代方案**：分散在各个配置解析器中 - 但会导致逻辑分散和重复

## Risks / Trade-offs

- **[风险] 验证逻辑可能遗漏某些边缘情况**
  - 缓解：通过单元测试覆盖常见场景，并在文档中说明已测试的场景

- **[风险] 输出警告信息可能被用户忽略**
  - 缓解：使用显眼的警告格式（如 `⚠️` 前缀），并在文档中强调配置优先级

- **[权衡] 验证可能增加启动时间**
  - 可接受：配置验证是轻量级操作，对性能影响极小
